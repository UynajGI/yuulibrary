    物理学家的软件概要#

    计算物理学的目标是攻克其他方法无法处理的问题，补充实验和理论研究，建立对其结果的信任，并探索超出其范围或分析特别困难的参数区域。对于计算量子物理学来说尤其如此，在该领域中，分析工具的应用面临着巨大挑战，例如在多体系统中。特别是当目标是进行顶尖学术研究时，方法和工具不仅仅是现有技术的简单应用：相反，它们被推到了极限，在需求和目标变化非常迅速的快速演变的场景中。确实，与开发软件的商业应用领域不同，关于软件应在何种情况下、使用何种资源执行何种功能的规范（软件需求规格说明）不能——也不可能——一劳永逸地精确定义，而是随着软件的使用加深对所应对问题的理解而不断演变。因此，在规划开发科学软件时，应该意识到，无论多么努力，最终都必须满足三个要求，即：

    R1 随着情况演变和相应需求的变化而修改软件；

    R2 处理大量数据；1

    R3 使用大规模和/或低效算法解决难题。

    鉴于上述三个需求 R1、R2 和 R3 如此普遍且无处不在，专业计算物理学家应当学会如何应对它们，就像学会求解积分或掌握其研究领域中的特定技术一样。事实上，上述三个需求要求采用过去几十年发展起来的某些策略，特别是：

    S1 软件工程：编写良好且灵活的软件，易于调试和扩展；

    S2 脚本化：尽可能将工作自动化，因为前处理和后处理所需的工作量将与生成数据本身相当；

    S3 优化与高性能计算：尽可能高效，并准备好在必要时使用“蛮力”。

    本章引导读者了解上述策略，这些策略应以最高效的方式用于攻克科学计算问题。要应对上述看似巨大的挑战，一个良好的出发点是以下优先级列表，任何编写科学软件的人都应尽可能遵循这些优先级 [120, 121]。这些优先级普遍适用于任何科学软件（其中大多数也适用于任何软件），然而，为清晰起见，下文将针对量子物理领域进行专门阐述。按优先级顺序排列，并附有明确的优先级动机：

    1. 正确性——要有用；

    2. 数值稳定性——要可信；

    3. 精确离散化——要描述自然；

    4. 灵活性——要可用；

    5. 效率——要做得漂亮！

    在本章剩余部分，我们将向读者介绍每个方面最重要的内容，这些内容在编写优秀的科学软件时应牢记于心。

    B.1 正确性#

    优秀的科学软件的基本要求显然应该是正确的，即它应按照所用编程语言、版本及编译器的要求，以正确的语法执行正确的指令序列。然而，即使是技术娴熟、经验丰富的程序员，编写出没有任何错误的几行代码的概率也基本为零。因此，必须制定并采取行动和策略来纠正这些不可避免的错误。第一个支持来自编译器，这是一个将源代码转换为低级代码的程序。通常，编译器能够识别语法错误，并且不断改进以涵盖更多可能的错误场景。然而，即使编译器运行无误并生成编译后的代码（即可执行程序），该程序仍可能因多种不同原因而崩溃，例如内存分配问题（程序试图写入未分配的内存位置）、“差一错误”（Off-by-one，例如递归索引从零开始而不是从一开始，或反之）。发现并纠正这些错误及其他错误就是“调试的艺术”，需要时间和经验。不过，可以采取一些好的实践来加速这一过程，下面简要介绍：

    – 使用打印（print）语句检查变量名，同时结合编译器选项，该选项允许选择用于调试的代码行（例如，ifort 编译器的 d 选项，以及在调试测试行的开头使用 d）：

    1 d PRINT * , " x 的值是 " , x

    这种组合是一种快速减少调试时间的方法，因为为此目的编写的代码行不会被删除，而是保留在那里，以便在出现另一个错误时重复使用。

    – 对于更复杂的检查，当需要编写多行代码和/或可在程序其他部分复用这些代码时，可以定义专用的子程序（SUBROUTINE）或函数（FUNCTION），并带有一个启用全局变量以及扩展标志和输出。此类代码最简单的示例可能采用以下形式：

    FUNCTION 检查点
2 IF (DEBUG)
3 PRINT * , " x 的值是 " , x
ENDIF
5 END FUNCTION

    – 最后，在代码复杂度达到一定水平以上时，强烈建议使用专业调试器，例如 GNU 调试器 GDB，甚至可以使用图形用户界面（DDD）[637,638] 辅助。

    使输出与输入兼容是一个好做法，它允许为程序自动生成复杂度递增的测试。例如，如果处理的是矩阵或张量，并且输入和输出样式匹配，则输出可以很容易地作为新输入使用，而无需进行耗时的格式处理。这种兼容性在数据采集阶段也极为有用，因为在许多情况下程序可能需要重新启动（例如，由于达到计算时限、需要额外检查收敛性等）。

    在程序中包含用于已知问题的“错误和异常”部分：即使不打算纠正它们，指出它们并明确包含一个测试来排除这些异常，将在调试期间节省大量时间。例如，如果在实现某个特定算法时排除了一个特殊情况（因为它需要当前不必要的额外工作），那么当同一段代码可能会被复用时（见后文第 B.4 节），并且可能恰好需要被排除的那个场景时，包含一个异常将非常有益：数月甚至数年后，发现该异常可能需要数小时的艰苦调试。

    – 针对随机输入进行测试：尽管随机输入通常不是病态的，因此可能无法涵盖所有情况（例如，随机矩阵永远不会是退化的 [133,639]），但随机测试允许执行大量的自动化测试。同样，大多数常见编程语言都有可用的自动化测试软件。

    如果按照前述做法，应该可以实现不同策略来高效调试。特别是可以采用增量测试（Incremental Testing），即在编写程序的同时测试和调试每个部分：分开调试代码的不同部分比一起调试要简单得多。然而，即使所有部分都没有错误，也不能保证在组合它们时不会出现某些错误。因此，另一个值得采用的策略是回归测试（Regression Testing），即将发现错误时产生错误的场景保存在一组自动化测试中。每次进行更改时，可以重新运行所有测试，只要这些测试经过精心设计和实现，应当能大大提高调试速度。还可以采纳极限编程（Extreme Programming）范式，该范式尤其强调在编写代码本身之前就设计和编写测试。这样，应该能够轻松地自动化这些测试并尽可能多地运行它们[640]。

    最后，即使编译器编译通过且程序运行不会崩溃，我们如何确信输出是正确的，即程序确切地计算了预期内容？为了增加对输出的信心（并且也有助于调试），在代码中包含一些正确性证明是有用的，例如：

    • 在代码的每个部分，即使是琐碎部分，也以注释形式（如果可能，以自动化检查形式）明确包含前置条件和后置条件，并在违反时给出清晰的信息。这会花费程序员一些时间，但在需要定位错误和/或代码返回不符合预期的结果时，将节省大量时间。前置条件的典型例子是变量的符号（例如，已知为正数）、向量的范数、向量、矩阵和张量的维度（通常应匹配）。后置条件可以体现在解的某些性质中（例如，特征向量归一化、函数单调性、某些特定参数的预期结果等），这些性质应成立，无论解本身的（未知）值如何。

    • 如果存在循环不变量（loop-invariant），则定义它们，因为它们可以为该部分代码的正确性提供归纳证明：检查递归索引 i 的给定值下的循环不变量，并检查如果它对 i 成立，那么对 i+1 也成立。

    • 在物理学以及适用的情况下，通常可以识别出一些平凡（例如，波函数归一化、系统能量等）和非平凡的运动常数，这些常数应受到监控。

    在探索新领域之前，应与其他程序结果以及问题解已知的情况（例如，无相互作用情况、单粒子或少数粒子、低维等）进行比较。

    B.2 数值稳定性#

    即使严格遵守了上述所有做法，代码也已启动并运行并通过了数值测试——因此可以安全地假设程序是正确的并执行了编写的算法——但可能仍然存在输出呈现无法信任的奇怪结果的情况。因此，即使程序在形式上正确，结果也应被丢弃：这通常是由于数值不稳定引起的，而数值不稳定主要是实数运算与浮点运算之间的差异造成的。检测和避免这种不稳定性可能并不容易，但在编写代码时应避免一些典型情况——如下所述[121]。

    • 切勿使用浮点变量测试相等性。代码行：
    1 REAL *8 x , y
2 x=y
3 PRINT * , ( x . e q . y )

    可能会输出 FALSE，因为相等性测试取决于架构和编译器的细节，以及对超出双精度变量有限精度的数字的检查。应始终使用两个浮点变量之间的差值进行检查，例如：

    1 REAL *8 x , y
2 err= 1d-14
3 IF ( a b s ( x - y ) . l t . e r r ) THEN
4 PRINT * , " x i s e q u a l t o y "
5 ENDIF
    • 同样，切勿使用浮点变量作为计数器。实际上，循环：
    REAL *4 x ,C
DO x = 0 . 0 , x<= 1 . 0 , x = x+ 1 . 0 / C

    每个周期会将变量 x 增加 1.0/C。因此，根据增量分数的有限精度表示所产生的舍入误差，该循环将循环 C+1 次或 C 次，从而剧烈且随机地影响计算结果。

    • 谨慎处理两个有限精度变量 a 和 b 相减的结果，当 $| a - b | \ll | a | , | b |$ 时。实际上，如果两个变量的舍入误差符号相反，可以证明差值 $c \equiv a - b$ 的相对误差与 $\Delta c / c \propto ( a + b ) / ( a - b )$ 成正比，这一比值容易轻易超出浮点精度范围2。这种错误在微积分中很常见，因为微分计算本质上是基于两个量之间的微小差值，例如导数的一阶近似为 $f^{\prime} ( x ) \sim \Delta f / \Delta x$。

    当自变量的可能取值跨越不同数量级时，计算 $\Delta x = x_{2} - x_{1}$ 可能会遇到上述问题，导致结果精度急剧下降，甚至产生完全随机的结果。

    最后，正如所有科学家所知，误差在算术计算过程中会传播，考虑到现代计算机能够执行的运算次数快速增长，必须仔细监控。特别是阶乘和重复乘法等运算需要谨慎处理。例如以下循环：

    1 REAL *8 f
2 f =F
3 DO k =0, x<N, k = k +1
   f = k * f

    当 $N \sim 20$ 时，会给出完全错误的结果。原因在于 $f_{0}$ 中的误差传播：由于有限精度数总可以假设为 $\hat{F} \equiv ( 1 {+} \epsilon ) F$，误差会迅速传播（尽管初始时很小而难以察觉），然后在几次迭代中爆炸，以看似不可预测的方式破坏结果。

    1 有趣的是，这里的“大数据”这个词与科学界中的含义不同，在科学界中，它指代包含大量条目的数据集，但通常具有结构化或分层结构。然而，在商业领域，“大数据”几乎总是指通常无结构的数据库集合，从中寻找隐藏的模式，即数据挖掘。

    B.3 精确离散化#

    通常，计算物理学家编写代码来模拟物理过程，这些过程大多数情况下可以用一组偏微分方程（PDEs）描述。在模拟PDEs时，为确保所开发程序提供有用信息，需执行的最后一项检查（在前述章节描述的检查之后）是模拟结果与物理现实之间的关系。事实上，模拟可能在数学上正确且数值稳定，但要发挥作用，它必须正确描述所研究现象。

    例如，PDE可以是薛定谔方程的一个特定实例，该方程是时间一阶PDE，可形式化表示为

    $$ \left\{\begin{array} {c} {\partial_{t} \psi ( x , t ) = \mathcal{L} \left[ \psi ( x , t ) \right]} \\ {\psi ( x , 0 ) = \psi_{0} ( x )} \end{array} , \right.\tag{B.1} $$

    其中$\partial_{t}$是对时间变量的偏导数，$\psi_{0} ( x )$表示系统的初始波函数，利乌维尔算子（Liouville operator）$\mathcal{L} \equiv - i \hat{H} / \hbar$由系统哈密顿量$\hat{H}$构成。求解式(B.1)的所有可能变体是本书的主要主题，其解（对于时间无关哈密顿量）可形式化地写为

    $$ \psi ( x , t ) = e^{\mathcal{L} t} \psi_{0} ( x ) \equiv U_{t} \psi_{0} ( x ) ,\tag{B.2} $$

    其中$U_{t}$是时序指数，即时间演化算子，它将零时刻的波函数映射到任意后续时刻$t$。求解数值问题等价于计算时间演化算子在初始波函数上的作用。在这一看似简单的操作中，有一个形式步骤的潜在后果不容低估：由于计算机模拟的浮点运算，连续时空$\boldsymbol{C} \equiv \left( \boldsymbol{x} , t \right)$必须映射为离散时空。也就是说，设定程序的精度并选择变量类型（如 REAL 4 或 REAL 8）时，我们用一个离散时空${\cal C} ~ \approx ~ {\cal C}_{\Delta}$（其中$\Delta \ \equiv \ ( \Delta x , \Delta t )$）来近似连续时空。典型的首选方案是使用线性均匀的时空网格，使得$x_{\ell} = \Delta x \cdot \ell$和$t_{n} = \Delta t \cdot n$，其中$\ell = 1 , \ldots , N_{x}$且$n = 1 , \ldots , N_{t}$，尽管也可以采用更复杂的方案，如自适应网格或有限元方法[128]。然而，无论网格选择如何，离散化通常会引入误差，使得

    B.4 灵活性#

    一旦前三个优先级得到满足（确保代码正确、数值稳定并能正确描述相关物理），所编写的软件就有可能被使用，有时甚至能持续多年。根据具体情况，它可能成为另一个更复杂软件的一部分，并最终可能成为一组用户、一个专家社区甚至非专家用户的标准。在编写计算机程序时，人们永远无法真正知道潜在的未来用户想用它做什么。这里的未来用户可能是一个月或一年后想重复使用已编写软件某些部分的同一个程序员；另一个发现你的代码有趣或有用的高水平程序员；一家计划将其集成到商业套件中的软件公司；程序员的上司；一名寻找软件来完成硕士或博士论文的学生；或者是大型软件项目或研究机构的IT主管，他们可能会将该程序转变为一个多用户、多用途研究软件的核心。

    每个软件都会经历所谓的软件生命周期，如图B.2所示：首先由需求或规格说明来描述，详细说明软件的特性及其要求的输出；然后将规格说明转化为软件设计，确保需求将被满足；接着用特定的编程语言将设计实现为代码。最后，对软件进行测试和调试，一旦足够可靠，便用于生成数据。最终，软件的使用会促使程序员更改规格说明和/或软件设计，从而闭合生命周期。对于更常用和著名的软件，生命周期甚至可能持续数十年，例如：

    操作系统（Windows, OSX, Linux等）、大型生产力套件（如Office）或科学套件（AUTOCAD, Matlab等）。需求或软件设计的变化可能由影响软件开发的不同方面触发，例如对新功能的需求（增加系统规模或多样性、新协议或数据采集等）、bug修复、新硬件、对其他软件的依赖以及目标用户的变化（例如从专家转向普通用户）。

    软件工程是一门致力于设计和实现软件的科学，因此它开发工具、概念和策略来应对软件开发及其生命周期（software lifecycle）所带来的挑战。复杂的策略和工具在编写小型单用户软件时可能并不相关，但随着软件复杂度的增长，它们变得愈发基础。由于现代科学几乎总是建立在复杂软件的性能之上，任何科学家在其学术生涯中都极有可能遇到大型复杂软件。因此，至少了解一些软件工程的基本概念非常重要。首要一点是，在编程时应进行通用化思考，留有余地，并且除非确实必要，不应使软件过于特化。如果因编写高度特化的代码部分而无法进行通用化，那么应编写一个将来可以轻松复用的库。总的来说，应力求在抽象性与编写代码所必需的具象实现之间取得良好平衡。如何做到这一点关乎经验，然而，应采用一些通用策略，以取得与开发者经验和技能兼容的最佳可能结果。采用这些策略，尽管在编程时需要额外的时间开销，但从长远来看很可能会带来效率的巨大提升。

    特别是，如前所述，软件开发者在不同抽象层次上执行不同的任务，从软件设计到单行代码的实现，再到程序各部分之间的接口。下面我们简要介绍一些良好的习惯，这可能有助于关注每个方面的重点并避免最常见的错误。

    软件设计 在规划新软件时，应依次考虑以下要点，并尝试回答相关问题：根据待开发软件的复杂程度，其中一些要点需要仔细审视，它们会影响整个软件开发过程，而另一些要点则可能被跳过。无论如何，花时间在这些要点上，再次强调，可能会在开发的后期阶段节省大量时间，并且必将有助于聚焦编程工作的重要方面，同时忽略或至少推迟次要方面。一个可能有用的检查清单是：

    1. 规范：#

    程序目标：软件的基本目标是什么？哪些功能将不被包含？

    性能：性能对项目的成功至关重要吗？程序将在单台机器、集群还是高性能计算架构上运行？

    输入/输出：输入和输出的格式是什么？它们的兼容性如何（例如，与其他程序的兼容性）？

    1. 数据结构与函数：需要哪些数据结构？是否需要定义带有自身函数的新数据结构？

    2. 库和语言：是否有现成的库可以执行某些所需任务？哪种编程语言提供了更好的工具？

    3. 协调：是否需要协调不同的编程人员和软件开发？使用哪些工具来支持他们？

    4. 可移植性：是否需要软件具有可移植性（即在不同平台和架构上运行）？如果需要，将如何实现？

    对于典型的科研项目，这些问题的答案往往相当直接：特别是对于大多数与量子物理相关的项目，诸如 FORTRAN、C++、Python 以及 LAPACK（线性代数包 [130]）数学库和一些版本控制与调试工具等标准工具就能胜任。然而，回答这些问题将迫使科学家更好地构建其工作中软件工程的部分，并避免一些可能破坏数月辛勤工作成果的常见错误。

    接口 一旦确定了软件结构，就需要对数据结构和所需函数进行编码：这通常意味着编写大量的代码片段（类定义、函数等），随后它们需要被接口化。这一操作绝非易事，而用于解决它的策略可能会影响最终结果。事实上，在此操作过程中，应尽可能利用以下方法：

    1. 数学抽象：在科学软件中，这通常是直接的，在量子物理学中，线性代数和复分析提供了描述感兴趣系统所需的工具。

    2. 信息隐藏：子程序和函数应仅通过其接口提供易于访问的信息，而不应暴露更多内容。这将避免不清晰的程序部分依赖关系，并使得调试和软件更新更加容易。

    3. 灵活性与模块化：抽象有助于提高软件的灵活性和模块化，这是编写良好、高效、可重用和有用软件的关键。例如，由于这些特性，BLAS（基本线性代数子程序，Basic Linear Algebra Subprograms [631]）能够并且确实被轻易地用作大多数现有科学软件的基础。

    下面，我们将简要描述一些组织工作的可能方法，并对其有效性进行评论 [121]。

    自上而下 (Top-down)：初始设计从高层开始实现，然后通过逐渐增加细节和具体性的步骤进行细化，直到软件可以运行为止。可以遵循这种策略，但它往往会降低最终软件的模块化程度，并增加调试过程的复杂性。

    自下而上 (Bottom-up)：首先定义数据结构及其操作的函数，然后编写高层代码，直到整个设计完成。这是一种有效的策略，但需要非常周密的规划，并且通常需要较高的编码技能才能最佳地利用。

    中间向外 (Middle-out)：可以采取一种折中策略，利用前两种策略的优点并降低其风险，但这需要一些勇气，因为有时需要倒退几步。然而，强制程序员细化之前完成的工作，往往能产生更易于调试且质量更高的代码。应遵循的步骤如下：

    1. 在高层确定总体指令；
    2. 确定实现最小可工作软件所需的组件；
    3. 编码实现最小可工作软件的组件（以其最小版本）；
    4. 在构建软件的同时进行测试，保留旧测试，更新它们并在更改后重新应用；
    5. 从接近底层的地方开始，构建复杂度递增的软件组件；
    6. 一旦软件的最小版本可以工作并经过调试，从第1步开始添加功能以达到最终目标。

    可以清楚地看到，这种策略将迫使程序员在推进过程中重新设计数据结构并重写部分代码。在编写良好、易于测试的代码中，这应该是可能的（尽管显然不是毫无痛苦），并且通过构建，最终将产生一个具有信息隐藏、灵活性和模块化的代码。

    编码 一旦软件设计完成并选择了策略，剩下的唯一事情就是编码本身。无论做出何种选择，编码都关乎知识、技能、耐心和经验。下面介绍一些良好实践，以引导新手掌握这门艺术的过程。

    1. 程序员中有一句名言：“过早优化是万恶之源”，这无需多言，如果说要补充的话，那就是为什么在本章中效率被放在最后一点的原因。
    2. 变量和函数的名称应仔细且恰当地选择。实际上，函数
      FUNCTION f ( x )
2 REAL 8 x , f
END FUNCTION
      其特点是名称简短、书写快捷，这显然有提高编写速度的优势。然而，如果这些名称被嵌入到数百个其他类似的名称中，那么它们的名称并不会提供任何关于其用途或含义的信息，程序员将很难记住它们全部。更危险的是，人们通常需要在数千行代码中自动查找变量和函数：自动搜索（或者更糟的是查找替换——这在进行代码重构时是常见做法）将会惨败。相反，同一个函数写成
      FUNCTION xsquare ( xvariable )
2 REAL 8 xsquare , xvariable
END FUNCTION
      这样会减慢编码速度而没有任何好处。因此，最好的解决方案是在这里给出的两个极端例子之间小心地取得平衡，这需要在定义变量名称时给予应有的关注。

    B.3 代码注释#

    注释代码对程序员来说总是一件困难的事情，然而出于上述原因，这是一个基本步骤。养成良好注释习惯的一个良好起点是查看专业库（例如 BLAS 库），并尝试复现它们的标准，这些标准允许用户从源代码编译出 HTML 格式的文档。然而，以下提供了一些指导原则：

    A. 注释接口，因为它们将是你最常需要的信息，例如：

    c NORMV( ) 计算向量 a(i) (i=1 到 n) 的范数
c
c a     输入数组
c n     数组长度
c norm= sqrt( sum a(i)^2 ) (flag=0)
c norm= 1 如果 n 小于或等于零 (flag=1)
c
      SUBROUTINE NORMV( a , n , flag )

    B. 声明循环不变量（loop invariants）和条件。

    C. 明确陈述异常行为的警告。

    D. 描述做了什么，而非如何做：实现的算法可以在代码文档中解释，或直接引用参考文献。

    E. 避免冗余的注释，它们会增加代码长度并浪费双倍时间：编写它们的时间，以及之后在寻找有用信息时阅读它们的时间。

    F. 保持注释的更新，因为过时的注释可能比完全没有注释更严重地拖慢调试和未来的代码重用。

    B.4 效率#

    代码效率是本章所列列表中优先级最低的一项，因为只有当所有先前的事项都正确处理完毕后，才值得考虑它。然而，它仍然是一个基本问题，可能会决定一个软件是顶尖的还是业余的，从而可能开启或阻碍进行前沿研究的可能性。软件复杂性是一个高度发展的理论，我们向感兴趣的读者推荐相关文献[127]。这里我们仅简要回顾两个关于复杂性不同定义的结果，它们对计算物理学有影响：

    算法复杂性（Algorithmic complexity）可能是最广为人知的复杂性定义，它根据运行算法所需资源（时间或内存）随输入大小 N 的缩放比例来对算法进行分类。这个定义除了对复杂性类别[127, 641]的基本介绍之外，还允许量化不同算法的性能，并区分困难算法（例如，与 N 呈指数或超指数缩放）和实用算法（多项式缩放，(N^{p})，其中 p 为低次幂）。

    平滑复杂性（Smoothed complexity）是最近引入的概念，它定义了在存在噪声的情况下，运行一个算法所需资源随输入大小 N 的缩放比例。这个看似微小的差异实际上非常重要，原因有二：第一，噪声在物理系统中是不可避免的，而且有限精度算术可以被视为一种有噪算术。第二，最近的研究表明，许多解决非常普遍问题的重要算法确实具有超多项式算法复杂性，但也具有多项式平滑复杂性[642, 643]。这意味着在算法复杂性方面效率低下的算法可以在实践中使用，例如用于求解线性规划问题的单纯形算法（simplex algorithm）[644]。

    无论选择何种复杂性度量，在大多数情况下，都有不同复杂性的算法来解决给定的问题。复杂性的差异通常体现在待编写算法的难度和长度上，通常具有更好缩放比例的算法涉及更长的编码。然而，对效率的追求总是倾向于选择缩放比例更好的算法，而不是较慢的算法，即使差异仅仅是从 (O \big( N^{2} \big)) 到 (O(N \log N))。此外，如果程序组织良好且编写得当，那么实现这两种算法都应该是容易的，将第一种作为快速启动和更高效算法的测试平台。

    其他有助于提高效率的做法包括：使用编译器优化标志（例如，ifort 编译器的 O# 和 fast 选项）、使用性能分析器（profiler）查找程序瓶颈（例如，gprof 或类似工具），以及仔细处理内存分配。

    实现高效计算的最后一步是利用并行架构和一般的高性能计算工具，同时使用国内外的超级计算机和大型计算集群 [126]。然而，这些额外的步骤可能需要巨大的投资，只有在探索了所有其他选项，而提高效率的唯一剩余可行步骤是投资于巨大的计算能力时，才应考虑这些投资。

    最后一点需要说明：在现代专业科学计算中，NIH（非我发明，Not Invented Here）综合征没有立足之地。因此，对于标准算法（例如线性代数）来说，没有理由自己编写库。投入时间学习如何使用现成的专业库（例如 BLAS、LAPACK 等）远比从头开始编写高效得多。事实上，通常“95% 的执行时间花费在 5% 的代码上”，而对于科学（量子模拟）软件，这些时间主要花费在基本的（线性代数）操作上。因此，最高效的策略是依靠该领域顶尖专家多年（有时是数十年！）的工作成果：在不投入类似精力和技能进行编码的情况下，改进他们代码的概率几乎为零（除非你已成为或立志成为开发这些库的少数顶尖专家中的一员）。
