# 金融中的交叉验证

## 7.1 动机

交叉验证（CV）的目的是确定 ML 算法的泛化误差，以防止过拟合。CV 是标准 ML 技术在应用于金融问题时失败的又一个实例。过拟合将会发生，而 CV 将无法检测到它。事实上，CV 将通过超参数调优助长过拟合。在本章中，我们将学习标准 CV 为什么在金融中失效，以及可以做些什么。

## 7.2 交叉验证的目标

ML 的目的之一是学习数据的一般结构，以便我们能在未来的、未见过的特征上做出预测。当我们在用于训练的同一数据集上测试 ML 算法时，不出所料地获得了惊人的结果。当 ML 算法被那样误用时，它们与有损压缩算法没有什么不同：它们能以极高的保真度概括数据，但预测力为零。

CV 将从 IID 过程抽取的观测分为两个集合：*训练集*和*测试集*。完整数据集中的每个观测属于且仅属于一个集合。这是为了防止一个集合向另一个集合泄露，因为那将破坏在未见数据上测试的目的。更多细节可在参考文献部分列出的书籍和文章中找到。

有许多替代 CV 方案，其中最流行之一是 k 折 CV。图 7.1 说明了 k 折 CV 执行的 *k* 次训练/测试划分，其中 k = 5。在该方案中：

1.  数据集被划分为 *k* 个子集。
2.  对于 *i = 1,...,k*
    1.  ML 算法在排除 *i* 的所有子集上训练。
    2.  拟合的 ML 算法在 *i* 上测试。

![](images/Image00063.webp)

**图 7.1** 5 折 CV 方案中的训练/测试划分

k 折 CV 的结果是一个 k×1 的交叉验证性能指标数组。例如，在二元分类器中，如果交叉验证精度超过 1/2，则认为模型学到了东西，因为那是我们抛均匀硬币将获得的精度。

在金融中，CV 通常用于两种场景：模型开发（如超参数调优）和回测。回测是一个复杂的主题，我们将在第 10-16 章中充分讨论。在本章中，我们将专注于模型开发的 CV。

## 7.3 为什么 K 折交叉验证在金融中失效

到目前为止，你可能读过不少金融论文，它们提供 k 折 CV 证据表明 ML 算法表现良好。遗憾的是，这些结果几乎可以确定是错误的。k 折 CV 在金融中失效的一个原因是，不能假设观测来自 IID 过程。CV 失败的第二个原因是，测试集在模型开发过程中被多次使用，导致多重检验和选择偏差。我们将在第 11-13 章重新审视这第二个失败原因。目前，让我们只关注第一个失败原因。

当训练集包含也出现在测试集中的信息时，就会发生泄露。考虑一个序列相关的特征 X，它与在重叠数据上形成的标签 Y 关联：

-   由于序列相关，X~t~ ≈ X~t+1~。
-   由于标签从重叠数据点导出，Y~t~ ≈ Y~t+1~。

通过将 t 和 t+1 放入不同集合，信息泄露了。当分类器首先在 (X~t~, Y~t~) 上训练，然后被要求基于观测 X~t+1~ 预测 E[Y~t+1~|X~t+1~] 时，即使 X 是无关特征，该分类器也更可能达到 Y~t+1~ = E[Y~t+1~|X~t+1~]。

如果 X 是预测性特征，泄露将增强已有价值策略的表现。问题在于无关特征存在时的泄露，因为这会导致错误发现。至少有两种方法可以降低泄露的可能性：

1.  从训练集中丢弃任何观测 i，其中 Y~i~ 是用于确定 Y~j~ 的信息的函数，且 j 属于测试集。
    1.  例如，Y~i~ 和 Y~j~ 不应跨越重叠期间（样本唯一性的讨论见[第 4 章](ch04.md)）。
2.  避免过拟合分类器。这样，即使发生一些泄露，分类器也无法从中获利。使用：
    1.  基础估计器的早停（见[第 6 章](ch06.md)）。
    2.  分类器的 bagging，同时控制冗余示例上的过采样，使各个分类器尽可能多样化。
        1.  将 `max_samples` 设置为平均唯一性。
        2.  应用序贯 bootstrap（[第 4 章](ch04.md)）。

考虑 X~i~ 和 X~j~ 在重叠信息上形成的情况，其中 i 属于训练集，j 属于测试集。这是信息泄露的情况吗？不一定，只要 Y~i~ 和 Y~j~ 独立。要发生泄露，必须 (X~i~, Y~i~) ≈ (X~j~, Y~j~)，而 X~i~ ≈ X~j~ 或甚至 Y~i~ ≈ Y~j~ 都不充分。

## 7.4 解决方案：净化 K 折交叉验证

减少泄露的一种方法是从训练集中清除所有标签在时间上与测试集中包含的标签重叠的观测。我将此过程称为「净化」（purging）。此外，由于金融特征通常包含表现出序列相关的序列（如 ARMA 过程），我们应该从训练集中消除紧随测试集中观测之后的观测。我将此过程称为「禁运」（embargo）。

### 7.4.1 净化训练集

假设一个测试观测，其标签 Y~j~ 基于信息集 Φ~j~ 决定。为了防止上一节描述的泄露类型，我们希望从训练集中清除任何标签 Y~i~ 基于信息集 Φ~i~ 决定的观测，使得 Φ~i~ ∩ Φ~j~ = ∅。

特别地，当两个观测 i 和 j 的标签 Y~i~ 和 Y~j~ 并发时（见[第 4 章](ch04.md)第 4.3 节），即两个标签都取决于至少一次共同随机抽取，我们将确定它们之间存在信息重叠。例如，考虑一个标签 Y~j~，它是闭区间 t ∈ [t~j,0~, t~j,1~] 内观测的函数，Y~j~ = f[[t~j,0~, t~j,1~]]（对记号有滥用）。例如，在三重障碍标记法（[第 3 章](ch03.md)）的上下文中，这意味着标签是索引 t~j,0~ 和 t~j,1~ 之间价格条跨越的收益符号，即 ![](images/Image00065.webp)。标签 Y~i~ = f[[t~i,0~, t~i,1~]] 与 Y~j~ 重叠，如果满足三个充分条件中的任何一个：

1.  t~j,0~ ≤ t~i,0~ ≤ t~j,1~
2.  t~j,0~ ≤ t~i,1~ ≤ t~j,1~
3.  t~i,0~ ≤ t~j,0~ ≤ t~j,1~ ≤ t~i,1~

### 7.4.2 禁运

对于净化无法防止所有泄露的情况，我们可以在每个测试集*之后*对训练观测施加禁运。禁运不需要影响测试集之前的训练观测，因为训练标签 Y~i~ = f[[t~i,0~, t~i,1~]]（其中 t~i,1~ < t~j,0~，训练在测试开始前结束），包含的信息在测试时间 t~j,0~ 时是可用的。换言之，我们只关心紧随测试之后的训练标签 Y~i~ = f[[t~i,0~, t~i,1~]]，t~j,1~ ≤ t~i,0~ ≤ t~j,1~ + h。我们可以通过在净化前设置 Y~j~ = f[[t~j,0~, t~j,1~ + h]] 来实现该禁运期 h。一个较小的值 h ≈ 0.01T 通常足以防止所有泄露，这可以通过测试增加 k → T 时性能是否不会无限改善来确认。图 7.3 说明了紧随测试集之后的训练观测的禁运。代码片段 7.2 实现了禁运逻辑。

## 7.5 Sklearn 交叉验证的缺陷

你可能认为像交叉验证这样关键的东西会在最流行的 ML 库之一中被完美实现。遗憾的是，情况并非如此，这是你必须始终阅读你运行的所有代码的原因之一，也是开源的有力论据之一。开源代码的众多优势之一是你可以验证一切并根据需要进行调整。代码片段 7.4 解决了两个已知的 sklearn bug：

1.  评分函数不知道 `classes_`，这是 sklearn 依赖 numpy 数组而非 pandas 序列的结果：https://github.com/scikit-learn/scikit-learn/issues/6231
2.  `cross_val_score` 会给出不同结果，因为它将权重传递给 fit 方法，但不传给 `log_loss` 方法：https://github.com/scikit-learn/scikit-learn/issues/9144
