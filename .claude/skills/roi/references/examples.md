# ROI 输出示例

## 示例 1：多书项目——有明确技术债

状态摘要：4 本书，validator 报告 28 issues（26 处 $$ 中文误包 + 2 处裸代码），构建通过，无阻断性问题。最近 commit 为新书格式化。

```
## ROI 建议

> 28 validator issues，全在 quant-finance 的 $$ 误包（26 处）和 rl-intro 裸代码（2 处）。

### 🥇 优先级 1：批量修 quant-finance ch03-07 的 26 处 $$ 中文误包
- **为什么**：整个 quant-finance 书的公式排版都被污染，读者看到的是中文碎片不是数学。26 处是同一模式，一个脚本可批量修。
- **影响**：I=4（6 个文件，全书近半章节）/ E=短
- **预估**：15 分钟（写脚本 5 分钟 + 跑脚本 5 分钟 + 验证 5 分钟）

### 🥈 优先级 2：给验证器加「$$ 内中文正文」检测规则
- **为什么**：这 26 处是人工发现的，下本书还会有。加规则后自动告警，每本书都受益。
- **影响**：I=5（永久复利，每本书受益）/ E=短（~10 行代码）
- **预估**：10 分钟

### 🥉 优先级 3：rl-intro ch05/06 裸代码加 fence
- **为什么**：代码没有高亮、没有复制按钮、# 注释被渲染成标题。2 个文件，4 处。
- **影响**：I=3（20 行代码）/ E=短
- **预估**：10 分钟

### 其他可考虑
- 首页样式微调（视觉优化，无功能问题，排在最后）
```

## 示例 2：干净项目——没有明显 Bug

状态摘要：4 本书，validator 0 issues，hugo build 通过（125 pages，1979ms）。所有已知问题已修完。

```
## ROI 建议

> 0 issues，构建通过。当前项目状态干净。

### 🥇 优先级 1：验证器加语义检查
- **为什么**：当前只做机械检查（$$ 配对、标题层级、fence 检测），不加语义检查的话 OCR 错误（如中文误包公式）只能肉眼发现。加一条规则，每本书永久受益。
- **影响**：I=5（复利）/ E=中
- **预估**：30 分钟

### 🥈 优先级 2：添加下一本书
- **为什么**：流程已验证（MinerU → 清洗 → 拆分 → Phase 4.5 → 格式化），加新书边际成本低。4 本书的 pipeline 经验已沉淀在 skill 和 memory 里。
- **影响**：I=3（一本新书）/ E=长
- **预估**：2-4 小时（取决于书的大小和 MinerU 质量）

### 其他可考虑
- 首页书架样式微调（视觉优化，无功能问题，排在最后）
- 更新 CLAUDE.md 新增的 validator 检查说明（已完成，无需重复）
```
