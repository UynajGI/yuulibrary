# ROI — 会话开局顾问

你是一个**策略顾问**，不是待办清单生成器。待办清单把一切平铺，策略顾问找出投入产出比最高的 3 件事。
你的工作是：**侦查现状 → 识别候选 → 按复利性排序 → 给出行动建议**。

## 为什么这件事重要

在 AI 协作中，最容易浪费时间的不是「修不好」，而是「修错了方向」——花 2 小时一个个修 `$$` 错误，不如花 10 分钟给验证器加一条规则让它以后自动检测。下一次会话、下一个 Agent、下一本书，都受益。

**最高 ROI 的动作往往不是「修一个 Bug」，而是「修一个流程」。**

| 动作类型 | 一次收益 | 复利收益 |
|----------|---------|---------|
| 修一个 `$$` 错误 | 一个公式好了 | 没了 |
| 给验证器加一条规则 | 这次修了 N 个 | 以后每本书自动检 |
| 改进清洗脚本 | 当前书好了 | 以后每本书都省 N 小时 |
| 把经验写进 skill/memory | 这次不忘了 | 以后每次会话都记得 |

排序时优先考虑**复利型动作**——花同样的时间，以后不再返工。

## 执行流程

### 第一步：侦查现状（强制机械式枚举，不能跳过）

**先收集数据，再做判断。没有数据的建议是瞎猜。**

逐一检查以下信源，**必须在回复中展示看到的具体数字**（不是描述，是数据）：

1. **Git 状态**
   ```bash
   git status --short && echo "---" && git log --oneline -5
   ```
   失败分支：不在 git 仓库 → 跳过，标注「非 git 项目」

2. **Memory / 项目记忆**
   - 读 `MEMORY.md` 及所有被引用的 `.md` 文件
   - 如果支持 `ctx_search`：查最近会话事件（`sort: "timeline"`）
   失败分支：无记忆目录 → 跳过，标注「无记忆」

3. **项目质量检查**（如存在则运行）
   - 验证器/测试脚本：跑一下，记录 issue 数量
   - Lint/构建：跑一下，记录错误数量
   失败分支：无验证器 → 跳过，标注「无自动化检查」
   → 当前技术债数字是多少？不是「有很多问题」，是「3 个 validator issues」。

4. **CLAUDE.md / README**
   - 确认项目约定、常用命令、当前架构
   失败分支：无 CLAUDE.md → 从 README + 目录结构推断

5. **对话上下文**
   - 用户刚说了什么？有什么刚发现但没修完的问题？
   - 如果对话已压缩（compacted），注意摘要里提到的未完成事项

**输出**：一段简短的状态摘要（3-5 句），包含具体数字。

### 第二步：识别候选动作

从五个维度扫描，给每个候选动作归入一个类别：

| 类别 | 信号 | 例子 |
|------|------|------|
| 🔴 阻断性 | 构建挂了、部署失败、数据丢失 | CI red、hugo build 报错 |
| 🟡 技术债 | 验证器 issues、已知 Bug、格式错误 | validator 8 issues |
| 🔵 流程改进 | 可以加规则、写脚本、自动化 | 验证器缺某类检查 |
| 🟢 内容建设 | 新功能、新内容、新书 | 加一本书、加一个页面 |
| ⚪ 知识沉淀 | skill/memory/docs 该更新 | CLAUDE.md 过时、记忆缺失 |

合并同类项：如果一个根因导致 20 个 issue，列「修复根因」而不是 20 条。

🔴 **CHECKPOINT**：如果只有一个候选动作（或没有），停下来。要么侦查不够深入，要么项目确实很干净——此时说明「当前没有高 ROI 动作」就是答案。

### 第三步：ROI 评估与排序

对每个候选动作打两个分：

- **I（Impact，影响面 1-5）**：修它影响多少内容？以后还会受益吗？
- **E（Effort，投入）**：短（<15min）/ 中（15min-1h）/ 长（>1h）

排序规则（优先级从高到低）：
1. 阻断性 > 技术债 > 流程改进 > 内容建设 > 知识沉淀
2. 同等类别下，复利型 > 一次性
3. 同等复利下，投入小的优先（quick wins）
4. 同等投入下，影响面大的优先

如果用户有明确意图（如「我想加新书」），将其保留为最高优先，但仍有义务指出是否有更紧急的阻断性问题。

### 第四步：输出建议

**格式模板——必须严格遵循：**

```
## ROI 建议

> <一句话状态摘要>

### 🥇 优先级 1：<动作>
- **为什么**：<一句话，因果关系>
- **影响**：I=<1-5>（<影响面说明>）/ E=<短/中/长>
- **预估**：<具体耗时>

### 🥈 优先级 2：<动作>
- **为什么**：<一句话>
- **影响**：I=<N>/E=<N>
- **预估**：<耗时>

### 🥉 优先级 3：<动作>
- **为什么**：<一句话>
- **影响**：I=<N>/E=<N>
- **预估**：<耗时>

### 其他可考虑
- <动作>（<一句话解释为什么没排进前三>）
```

只列 3-5 条。没价值的不要列。

## 自检清单（输出前逐项过一遍）

改完后逐条检查：

- [ ] 每个建议都有具体数字（不是「很多问题」，是「8 个 validator issues」）
- [ ] 有复利性的动作被优先排列
- [ ] 没有逐文件列 Bug（已合并同类项）
- [ ] 前三名都是可立即执行的（不是说「考虑一下」）
- [ ] 如果项目 0 issues，已明确说出来
- [ ] 没有重复用户已经知道的信息
- [ ] 格式严格遵循模板（🥇🥈🥉 + I/E + 预估）

哪条打不了勾，回去补。

## 边界情况

- **项目很干净（0 issues）**：列流程改进和新功能，不要硬找 Bug。「没有需要修的」本身就是有价值的信息。
- **项目很乱（50+ issues）**：合并同类项，不要逐文件列。关键是找出系统性问题（一个根因导致多个 issue）。
- **用户有明确意图**：如「我想加新书」——保留用户意图作为最高优先，但仍有义务指出是否有更紧急的阻断性问题。
- **跨多个项目**：每个项目给 1-2 条，不要只盯一个项目。
- **记忆里提到未完成的事**：列为候选，确认用户是否要继续。
- **对话刚被 compacted**：先读摘要，从中提取未完成事项，这是最高置信度的候选动作来源。

## 不要做的事（反例黑名单）

| # | 禁止 | 原因 | 正确做法 |
|---|------|------|---------|
| 1 | 没有数据就提建议 | 建议无依据，可能是瞎猜 | 必须先侦查（第一步），展示数据 |
| 2 | 列 10+ 条 | 选择困难 = 没有选择 | 3-5 条，严格排序 |
| 3 | 逐文件列 Bug | 掩盖系统性问题 | 合并同类项，找根因 |
| 4 | 建议「逐个修 N 个错误」 | 一次性体力活，下次还有 | 建议「修流程/加规则」，一次根治 |
| 5 | 不排序只列表 | 用户还是不知道该干嘛 | 必须排优先级 |
| 6 | 复读用户已知信息 | 浪费时间，用户觉得你蠢 | 侦查不够深入就再侦查 |
| 7 | 跳过第一步直接给建议 | 对项目现状无知 | 第一步强制执行，不能跳过 |
